Read the file first. The current implementation uses `SchedulerBinding.instance.addPostFrameCallback`
to defer provider mutation from `deactivate()`. This is insufficient — `addPostFrameCallback` runs

Replace BOTH `SchedulerBinding.instance.addPostFrameCallback` calls with `scheduleMicrotask`:

```dart

// Replace:
// SchedulerBinding.instance.addPostFrameCallback(

// With:

```

Apply this replacement in both locations in saveFocusKey() where the key is stored
(one for the directKey case, one for the found ancestor key case).

Remove the `import 'package:flutter/scheduler.dart';` import (no longer needed).

   ```dart
   /// D-pad Left at the left edge of a zone moves to the nearest zone to the left.
   /// Returns true if focus was moved (caller should mark as handled).
   bool _handleArrowLeft() {
     final escalation = ref.read(focusEscalationProvider);
     final primaryFocus = FocusManager.instance.primaryFocus;
     if (primaryFocus == null) return false;

     final railNode = escalation.railNode;
     final sidebarNode = escalation.sidebarNode;
     final sourceSelectorNode = escalation.sourceSelectorNode;

     // Already in rail — nowhere left to go.
     if (railNode != null && railNode.hasFocus) return false;

     // In sidebar — go to rail.
     if (sidebarNode != null && sidebarNode.hasFocus) {
       if (railNode != null && railNode.canRequestFocus) {
         railNode.requestFocus();
         return true;
       }
     }

     // In source selector (Zone B.5) — go to sidebar if present, else rail.
     if (sourceSelectorNode != null && sourceSelectorNode.hasFocus) {
       if (sidebarNode != null && sidebarNode.canRequestFocus) {
         sidebarNode.requestFocus();
         return true;
       }
       if (railNode != null && railNode.canRequestFocus) {
         railNode.requestFocus();
         return true;
       }
     }

     // In content (Zone C) — cross to nearest left zone.
     // Flutter's default traversal handles intra-content Left movement.
     // We only cross zones when the primary focus is a direct descendant of the
     // content area and there is a zone to the left. Use FocusManager to check
     // whether the current focus node is NOT within rail/sidebar/sourceSelector
     // (i.e. it is in content), then request the nearest left zone.
     final bool inRail = railNode != null && railNode.hasFocus;
     final bool inSidebar = sidebarNode != null && sidebarNode.hasFocus;
     final bool inSourceSelector =
         sourceSelectorNode != null && sourceSelectorNode.hasFocus;
     if (!inRail && !inSidebar && !inSourceSelector) {
       // Focus is in content. Cross to nearest left zone.
       if (sourceSelectorNode != null && sourceSelectorNode.canRequestFocus) {
         sourceSelectorNode.requestFocus();
         return true;
       }
       if (sidebarNode != null && sidebarNode.canRequestFocus) {
         sidebarNode.requestFocus();
         return true;
       }
       if (railNode != null && railNode.canRequestFocus) {
         railNode.requestFocus();
         return true;
       }
     }

     return false;
   }
   ```

   NOTE: This handler fires ONLY when AppShell's KeyboardListener receives the event.
   Flutter routes arrow keys to the focused widget first (FocusTraversalPolicy handles
   intra-zone movement). This handler is the fallback when traversal cannot proceed
   within the current zone (Flutter returns KeyEventResult.ignored). Because AppShell
   is an ancestor, it sees the event only after descendants have had a chance to handle
   it. For Left arrow specifically, content widgets (lists, grids) handle intra-zone
   movement first; this handler runs when they ignore the event (at the left edge).
