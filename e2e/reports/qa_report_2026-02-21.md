# Comprehensive QA Report — CrispyTivi Web

**Date**: 2026-02-21
**Tester**: Claude Code (automated Playwright crawl + manual screenshot analysis)
**Build**: `flutter build web --release` (CanvasKit renderer)
**Server**: `npx serve build/web -l 3000`
**Viewport**: 1280x900 (desktop), plus responsive checks at 360x800, 840x600, 1920x1080

---

## Executive Summary

The Playwright coordinate-based crawl captured **32 screenshots** across
profile selection, settings, live TV, program guide, home, and responsive
viewport resizes. The crawl captured **5 runtime errors** and
**4 WebGL warnings**. Key findings:

- **P0 — Runtime crash**: Null-check operator crash during keyboard Tab
  navigation (focus management bug)
- **P1 — Navigation trap**: Sub-pages (Media Servers, etc.) remove the
  sidebar entirely, trapping the user with only a back button
- **P1 — First-run routing**: After profile selection with no sources
  configured, app goes to Settings and auto-opens "Add Xtream Codes"
  dialog unprompted
- **P2 — Keyboard nav broken**: Tab key triggers repeated runtime errors
  on Home screen; Escape key does not close sub-pages
- **P2 — Dialog UX**: Xtream Codes form uses placeholder-only labels,
  no persistent input labels

## Test Methodology

### Approach

Flutter CanvasKit renders everything into a `<canvas>` element.
Semantics-based selectors (`getByText`, `aria-label`) do NOT work
reliably in release builds. The first crawl attempt using semantic
selectors failed — all 17 screenshots were identical (stuck on profile
screen).

**Solution**: Created `manual-crawl.spec.ts` using pixel-coordinate
clicking against known layout positions. This approach successfully
navigated the app and captured meaningful screenshots.

### Test Flow

1. App load → Profile selection screen
2. Click Default profile card at (590, 460)
3. Navigate via sidebar icon rail (x=28) to each tab
4. Interact with settings items
5. Test keyboard navigation (Tab, Arrow, Enter, Escape)
6. Test right-click and long-press context menus
7. Resize viewport to mobile (360x800), tablet (840x600),
   TV (1920x1080), and back to desktop (1280x900)

## Issues Found

### P0 — Critical

#### BUG-001: Null check operator crash during keyboard Tab navigation

- **Screen**: Home screen (after Tab key presses)
- **Severity**: P0 — Runtime crash (non-fatal but error-spamming)
- **Screenshots**: 023-025 (app was on Media Servers by then)
- **Console output**:
  ```
  [08:01:26.897] Null check operator used on a null value
      at Object.bWP (main.dart.js:35340:3)
      at Object.bWQ (main.dart.js:35316:5)
      at aVT.a7v (main.dart.js:138360:19)
      at Object.bN6 (main.dart.js:35272:19)
      at aVT.qJ (main.dart.js:138051:5)
      ...
  ```
- **Followed by**: 5 `PAGE_ERROR: Cannot read properties of null
  (reading 'toString')` errors at ~650ms intervals
- **Root cause**: Focus management code calls `.toString()` on a
  null widget/element during Tab key focus traversal. Likely in
  `FocusWrapper` or the focus traversal policy.
- **Impact**: Console error spam. May cause focus to get lost or
  skip elements. Does not crash the app but indicates broken
  keyboard/remote navigation.
- **Action**: Debug in Chrome DevTools with source maps. Check
  `FocusWrapper.onFocusChange` and `FocusTraversalPolicy` for null
  access patterns.

### P1 — High

#### BUG-002: Sub-pages remove global navigation sidebar

- **Screen**: Media Servers, Connect Jellyfin, and other Settings
  sub-routes
- **Severity**: P1 — Navigation dead-end
- **Screenshots**: 012-032 (all stuck on Media Servers)
- **Description**: When navigating to a Settings sub-page (e.g.,
  "Browse Media Servers"), the entire nav rail sidebar disappears.
  The user is left with only a back arrow (←) and the page title.
  There is no way to navigate to other top-level tabs without going
  back first.
- **Impact**: Users who enter sub-pages lose all global navigation.
  Keyboard-only users (TV remotes, accessibility) may struggle to
  find the small back button. The crawl bot was completely trapped.
- **UX Standard Violated**: Nielsen's heuristic #3 (User control and
  freedom) — users should always be able to leave an unwanted state.
- **Fix options**:
  1. Keep the icon rail visible on sub-pages (preferred, matches
     TiviMate behavior)
  2. Add a breadcrumb trail
  3. Ensure Escape key navigates back from sub-pages

#### BUG-003: First-run routes to Settings with auto-open Xtream dialog

- **Screen**: After profile selection (first run, no sources)
- **Severity**: P1 — Confusing onboarding UX
- **Screenshots**: 003-005
- **Description**: After selecting the Default profile for the first
  time (no playlist sources configured), the app navigates to the
  Settings screen instead of Home. Additionally, the "Add Xtream
  Codes" dialog auto-opens without user action.
- **Impact**: New users are immediately confronted with a technical
  form (server URL, username, password) instead of a welcoming home
  screen. Users who don't use Xtream Codes are confused.
- **UX Standard Violated**: Nielsen's heuristic #7 (Flexibility and
  efficiency of use) — the first-time experience should guide, not
  overwhelm.
- **Fix options**:
  1. Navigate to Home after profile selection. Show a banner/card
     on Home: "No sources configured — Add a playlist in Settings"
  2. If redirecting to Settings, show a setup wizard instead of the
     raw Settings page
  3. Remove auto-open of Xtream dialog — let the user choose which
     source type to add

### P2 — Medium

#### BUG-004: Keyboard Escape key does not navigate back from sub-pages

- **Screen**: Media Servers and other sub-pages
- **Severity**: P2 — Keyboard/remote navigation gap
- **Description**: Pressing Escape on a sub-page (e.g., Media Servers)
  does not navigate back. The user is trapped unless they can
  click/focus the back button. The crawl test pressed Escape multiple
  times (during settings item clicks) but never escaped the sub-page.
- **Impact**: D-pad/remote users on Android TV and Fire TV cannot
  navigate back using the standard "Back" button mapping.
- **Fix**: Add `WillPopScope` / `PopScope` handling or intercept
  Escape key at the sub-page scaffold level to call `Navigator.pop()`.

#### BUG-005: Xtream Codes dialog uses placeholder-only labels

- **Screen**: Settings → Add Xtream Codes dialog
- **Severity**: P2 — Usability
- **Screenshots**: 003-005
- **Description**: The "Add Xtream Codes" dialog has four input fields
  (Name, Server URL, Username, Password) that use only placeholder
  text as labels. When a field is focused and the user starts typing,
  the placeholder disappears and there is no persistent label above
  the field to remind the user what it's for.
- **UX Standard Violated**: Material Design guidelines — input fields
  should have persistent floating labels, not just placeholders.
- **Fix**: Use `InputDecoration(labelText: ...)` instead of or in
  addition to `hintText`. This applies to all source-add dialogs
  (M3U, Xtream, Stalker Portal, EPG URL).

#### BUG-006: Quick Access cards link to unimplemented features

- **Screen**: Home screen
- **Severity**: P2 — Feature gaps
- **Screenshot**: 011
- **Description**: The Home screen Quick Access section shows cards
  for "Multi View", "DVR", "Cloud Storage" which are placeholders
  for features not yet implemented. Clicking these may lead to empty
  screens or error states.
- **Impact**: Users see features advertised that don't work, reducing
  trust and polish perception.
- **Fix**: Either implement the features, hide unimplemented cards,
  or show a "Coming Soon" badge/toast when tapped.

### P3 — Low

#### BUG-007: WebGL GPU stall warnings during rendering

- **Severity**: P3 — Performance (cosmetic)
- **Console output**:
  ```
  [.WebGL-0xff400e68e00]GL Driver Message (OpenGL, Performance,
  GL_CLOSE_PATH_NV, High): GPU stall due to ReadPixels
  ```
  (4 occurrences, self-silencing)
- **Description**: NVIDIA driver-level performance warnings during
  CanvasKit WebGL rendering. Common with GPU-accelerated 2D canvas
  operations. Does not affect functionality.
- **Action**: No fix needed. Suppress in test log filtering.

#### BUG-008: Profile card icons are generic placeholders

- **Screen**: Profile Selection
- **Severity**: P3 — Visual polish
- **Screenshots**: 001-002
- **Description**: The Default profile uses a generic purple person
  icon. The "Add Profile" card uses a basic `+` symbol in a dark
  gray box. These don't match the premium dark glassmorphism
  aesthetic of the rest of the app.
- **Fix**: Design custom profile avatars or use Material 3 avatar
  components with more visual appeal. Consider adding avatar
  selection (color, icon, or image).

#### BUG-009: Empty state inconsistency across screens

- **Screens**: Live TV, Program Guide, VODs, Home
- **Severity**: P3 — UX consistency
- **Description**: Empty states vary in quality:
  - **Live TV** (008): "No channels found / Add a playlist source
    in Settings" — good, but no action button
  - **Program Guide** (010): "No channels found" with a "Refresh"
    button — has action button but wrong action (refresh won't help
    if no sources)
  - **Home** (011): Shows Quick Access cards — no empty state
    messaging about missing sources
- **Fix**: Standardize empty states across all screens:
  - Consistent icon + message + CTA button
  - CTA should link to Settings > Add Source
  - Guide "Refresh" should become "Add Source" when no sources exist

## Playwright Test Infrastructure Notes

### Semantic selectors don't work in CanvasKit release builds

The `comprehensive-crawl.spec.ts` test uses Flutter semantic selectors
(`getByText`, `aria-label`). These rely on the Flutter accessibility
overlay (`flt-semantics` elements). In release builds with CanvasKit,
these elements are either not present or not reliably interactive.

**Recommendation**: All Playwright E2E tests for this app MUST use
coordinate-based clicking (`page.mouse.click(x, y)`). The
`manual-crawl.spec.ts` approach is the correct pattern.

### Nav rail coordinate mapping needs update

The test assumed 5 nav items but the app has 9. Correct icon-only
rail positions (at 1280x900 desktop):
